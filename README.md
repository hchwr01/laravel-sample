## ToDoリスト (/laravel/todos)
Laravel (PHP/MySQL)を用いて、以下の要件を満たす ToDo リストシステムの構築を実施。

### 概要
- やるべき作業（以下 ToDo）をリストとして管理し、閲覧・登録・更新・削除・完了管理ができる Web アプリケーションを作成。
- ブラウザから操作できる Web ページとして実装。

### 前提条件
- 単一ユーザー利用 を前提。
- 認証・ログイン機能の実装は不要。
- 安全性・保守性を意識した実装。

### 機能要件
- ブラウザから URL を入力することで ToDo リストの Web ページにアクセスできる。
- ページ表示時に、過去に登録された ToDo の一覧が表示される。
- 新しい ToDo を登録できる。
- 登録済み ToDo の内容を修正できる。
- 登録済み ToDo を削除できる。
- 登録済み ToDo を完了状態に変更できる。
- 上述の機能にページ遷移を伴わない、シングルページアプリケーション(SPA)であること。

### 技術要件
- 言語：PHP
- データベース：MySQL
- フレームワーク：Laravel
- ToDo データは MySQL で管理すること

### SPA要件
- ページ遷移を行わず、ページリロード無しで CRUD 操作が完結すること。
- Ajax / fetch / axios 等の利用。
- フロントエンドフレームワーク（Vue / React 等）の使用は 任意。
- SPA の実装品質やフロントエンドの作り込みは 必須条件ではない。

### 開発所感
SPA要件を満たすためにLivewire4を選びました。  
理由としてPHPを活かした実装が可能なこと、Laravelリソースを十分に活かせることが理由です。

Livewireコンポーネントクラスをコントローラーに見立て論理ロジックをFormに逃しています。  
Controllerは薄くし、ActionやRepositoryに論理ロジックを逃がすモダンな手法を踏襲するためです。

また、要件にある「安全性・保守性を意識した実装」のため、エラーハンドリングを実施しています。

データベースエラー等の障害に備え、登録処理等は例外キャッチ（\Throwable）によるログ出力を整えましたが、Laravel/Livewire特有のバリデーション例外（ValidationException）まで横取りしていました。

これに対し、バリデーション例外のみを明示的に識別して再スルー（throw）する制御を加えることで、裏側のログ監視と、500エラーにならないユーザーフレンドリーなエラー表示（Livewire標準のUI挙動）を実現しました。

### バイブコーディングに関して
今回のAIはローカルLLM（ollama / deepseek-coder-v2:16b）です。  
エージェント運用ではなくコード補完のみに使用しました。

実装されるコードに対する著作権が不明瞭なこと、なぜその実装が必要だったかの説明が難しくなること、レビュー／リファクタリングに実際のコーディングよりも大きな負担があることが主な理由です。
